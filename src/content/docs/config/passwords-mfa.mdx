## Password policy

Rules new and changed passwords must satisfy.

- **Recommended:** at least `length(12)` plus `notUsername` and `notEmail`; add a breach
  check (`Not recently used`, and a HaveIBeenPwned-style rule where available).
- **Ask the assistant:** *"Require a minimum password length of 12 and block the username for acme-prod."*
- **Keycloak:** Authentication → **Policies → Password policy**.

<Aside type="caution" title="Why it matters">
With no policy, users can set trivial passwords. The [Analyzer](/docs/config/) flags a
missing policy as **high** and a policy without a length rule as **medium**.
</Aside>

## One-time passcodes (TOTP)

Time-based OTP from an authenticator app (Google Authenticator, 1Password, etc.).

- **Recommended:** required for admin/privileged pools; optional-but-encouraged for consumer.
- **Ask the assistant:** *"Require OTP for acme-prod."*
- **Keycloak:** Authentication → **Required actions → Configure OTP**, and the OTP policy under
  Authentication → **Policies → OTP policy**.

## WebAuthn passkeys

Phishing-resistant passwordless sign-in with Touch ID / Face ID / Windows Hello / a security key.

- **Status:** on the [roadmap](/docs/roadmap/) — passkeys (WebAuthn/FIDO2) as a first-class,
  console-configurable option. Keycloak supports WebAuthn today; we're building the guided setup.
- **Why:** phishing-resistant MFA for SOC 2 and fewer password-reset tickets.

## Step-up authentication

Require stronger authentication only for sensitive actions (e.g. changing payment details),
rather than for every login.

- **Keycloak:** conditional authenticators in a custom **Authentication flow** keyed to an ACR
  level of assurance.
- See [Advanced Security](/docs/security/) for adaptive/step-up patterns.
